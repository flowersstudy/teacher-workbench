# 老师工作台上线前差距清单

## 1. 当前判断

- 如果只做 **公司内部小范围试用 / 内测**，项目已经接近可用。
- 如果要做 **团队长期稳定使用**，当前还不建议“全功能上线”。
- 结论：建议采用 **MVP 先上线、非核心功能先隐藏** 的方式推进。

### 当前可上线的 MVP 范围

- 老师登录 / 注册
- 学员列表
- 学员详情基础信息
- 学员备注 / 标记
- 文字聊天（含实时 WebSocket）
- 待批改列表
- PDF 批改
- 日历基础增删改查

### 当前不建议承诺上线的功能

- 总览大盘
- 排课 / 调课
- 学员分配 / 题目分配 / 考试分配
- 投诉闭环
- 私聊
- 群成员管理
- 聊天图片 / 文件 / 语音

## 3. P0：上线前必须先做

> 目标：让系统可以“真实使用”，同时避免团队点到假功能。

- 总览页：`src/pages/TeacherWorkbench/components/RightPanel/OverviewView.tsx`
- 排课页：`src/pages/TeacherWorkbench/components/RightPanel/SchedulingView.tsx`

- 老师端页面已切到真实接口 / seed 数据，旧前端演示数据已清理
- 上线后会出现“页面能打开，但数据不是真实数据”的问题
- 这种问题比功能暂时隐藏更伤信任

### P0-2 先关闭聊天中的非文字消息能力

#### 当前状态

- 文字消息已接入真实后端

- 贴纸上传还留有 TODO：`src/pages/TeacherWorkbench/components/RightPanel/ChatView.tsx`

- 上线前只保留文字发送
- 图片、文件、语音按钮先隐藏或置灰

### P0-3 做一轮最小上线验收

## 8. 粗略排期建议

### 方案 A：最快上线

- `第 1 天`：完成 P0 清理，去掉旧前端演示数据入口，完成内测部署
- `第 2 ~ 5 天`：完成 P1 清理，补异常学员、分配、成员管理、详情/批改真实数据化

### 方案 B：更稳妥

- `第 1 ~ 2 天`：完成 P0 + 联调测试
- `第 3 ~ 7 天`：完成 P1
- `第 2 周`：决定是否做 P2

---

## 9. 最终建议

- 现在 **不是不能上线**
- 但建议你按 **“MVP 先上线”** 的方式推进
- 不要把仍未接好真实接口的模块一起开放给团队
- 最现实的做法是：
  - 先上线真实可用的核心链路
  - 先隐藏假的和半成品功能
  - 再按 P1 / P2 补完整
